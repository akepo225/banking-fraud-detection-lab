# PRD: Banking Fraud Detection Lab v0.2

Status: draft for human review.

## Problem Statement

v0.1 proves that the lab can generate deterministic synthetic data, load it into
SQLite, model the **Alert lifecycle**, run baseline notebooks, and validate core
quality gates. The next problem is foundation durability. Later releases will add
deeper **Private-banking fraud detection**, **Digital-banking fraud detection**,
case interpretation, graph features, governance, and production patterns. Those
releases need a shared **Realistic synthetic data model** that is explicit,
documented, deterministic, and easy to extend through **Progressive data views**.

Without v0.2 hardening, later modules risk copying assumptions into notebooks,
inventing incompatible views, or changing the meaning of **Partner**, **Client**,
**User**, **Banking relationship**, **Detection pattern**, and lifecycle labels.

## Solution

Deliver a shared foundation release that strengthens schema documentation,
generator configuration, temporal realism, SQL foundations, data-quality
validation, and learner-facing setup. v0.2 should make the canonical data model
obvious before the curriculum branches into richer Alpine Crest Private Bank and
NovaBank Digital modules.

This release should add depth to the foundation, not many new fraud scenarios.
The main deliverable is a stable platform that later PRDs can depend on.

## v0.1 State Assessment

v0.1 already delivered the minimum public-ready foundation: deterministic tiny
sample data, a generator, SQLite loading, an explicit **Alert lifecycle**, baseline
private-banking and digital-banking notebooks, alert-aware evaluation utilities,
seeded case and regulatory notes, publication guardrails, and CI quality gates.

v0.2 should fill the foundation gaps that remain after v0.1:

- ERD-style schema diagrams and stronger module-specific view documentation.
- Explicit tiny, small, medium, and large generator scale profiles.
- More realistic temporal semantics for KYC risk, assignments, user
  authorization, account status, alerts, cases, and outcomes.
- Broader SQL foundations beyond the first representative examples.
- A generated data-quality report.
- Optional Python, pandas, SQL, and sklearn refresh material that uses the same
  synthetic data but stays outside the required core path.

## User Stories

1. As a learner, I want to understand the canonical data model, so that later fraud modules feel connected rather than arbitrary.
2. As a learner, I want **Progressive data views**, so that I can start simple and reveal complexity deliberately.
3. As a learner, I want clear ERD-style documentation, so that joins across **Partner**, **Client**, **User**, **Banking relationship**, accounts, alerts, and cases are explainable.
4. As a learner, I want deterministic small and medium datasets, so that I can reproduce notebook outputs on a laptop.
5. As a learner, I want more realistic effective dates and status fields, so that time-aware fraud questions make sense.
6. As a learner, I want SQL exercises for joins, windows, cohorts, alert queues, and feature extraction, so that SQL remains first-class.
7. As a learner, I want an explicit data-quality report, so that I can tell generated realism from generator defects.
8. As a maintainer, I want schema and data-dictionary checks, so that docs stay aligned with generated outputs.
9. As a maintainer, I want generator scale profiles, so that later notebooks can request tiny, small, medium, or large data consistently.
10. As a future module author, I want stable view contracts, so that v0.3 and v0.4 can add track depth without changing foundations.
11. As a reviewer, I want deprecated or ambiguous terms to be absent, so that public-facing language remains coherent.
12. As a learner who needs refreshers, I want optional warm-up notebooks using the same synthetic data, so that I can prepare for the core modules without creating a separate beginner track.

## Implementation Decisions

- Treat v0.2 as a shared platform release, not a track release.
- Keep SQLite as the required database path and keep other engines out of scope.
- Add or refine ERD diagrams and module-specific view documentation.
- Define scale profiles for tiny, small, medium, and large generated datasets.
- Improve temporal fields for KYC risk, relationship-manager assignment, user authorization, account status, alert creation, case opening, case closure, and outcomes.
- Keep **Detection pattern** metadata reusable enough for later case, graph, governance, and production releases.
- Make **Progressive data views** explicit artifacts with documented learner purpose.
- Add optional warm-up notebooks for Python, pandas, SQL, and sklearn only if they
  reuse the canonical synthetic data and remain outside the required module
  sequence.
- Keep protected answer keys out of learner-facing outputs by default.
- Prefer additive schema changes and documented migrations over rewrites.

## Testing Decisions

- Test deterministic generation for each supported scale profile that is practical in CI.
- Test schema contracts, data-dictionary alignment, nullability, primary keys, foreign keys, and view columns.
- Test temporal invariants, including effective-date ordering and lifecycle ordering.
- Test SQLite loading for generated views and representative SQL examples.
- Test data-quality report outputs against expected ranges instead of brittle row-level labels.
- Smoke-test optional warm-up notebooks if they are added.
- Keep medium dataset performance checks local or optional if they are too slow for CI.
- Continue notebook smoke tests for foundations and ensure v0.1 notebooks still run.

## Out Of Scope

- Deep Alpine Crest Private Bank feature engineering.
- Deep NovaBank Digital feature engineering.
- Graph analytics beyond documenting relationships needed later.
- Interpretability, model-risk templates, and production monitoring.
- External databases or production infrastructure.
- New public-release decision work.
- A full beginner programming curriculum.

## Cross-Version Links

- v0.4 depends on v0.2 for **Client** and **User** separation, session-ready temporal concepts, and shared lifecycle semantics.
- v0.5 depends on v0.2 for case and regulatory notes that can point to stable entities and views.
- v0.6 through v0.8 depend on v0.2 for stable graph nodes, model inputs, monitoring tables, and quality checks.

- The PRD keeps v0.2 focused on foundations rather than track implementation.
- All later dependencies are additive and compatible with the v0.1/v0.2 data model.
- The PRD preserves **Partner**, **Client**, **User**, and **Banking relationship** meanings.
- The PRD keeps SQLite first-class and avoids unnecessary infrastructure.
- The PRD has explicit tests for docs, schema, temporal logic, and generated data quality.
