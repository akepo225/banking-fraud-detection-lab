# PRD: Banking Fraud Detection Lab v0.7

Status: draft for human review.

## Problem Statement

By v0.6, learners can build features, cases, and graph evidence. They also need
to explain fraud model behavior, choose thresholds, document limitations, and
write governance-ready summaries. Without a dedicated release, interpretability
and model risk may remain scattered across notebooks and fail to teach why model
outputs should be used cautiously in the **Alert lifecycle**.

## Solution

Add a dedicated interpretability, governance, and model-risk layer. v0.7 should
teach feature importance, threshold selection, alert-capacity tradeoffs,
false-positive review, drift and stability concerns, model documentation, and
governance memos for both Alpine Crest Private Bank and NovaBank Digital.

v0.7 also owns the deepening of the v0.1 `03_alert_governance` notebook. The
early `03` module should remain the first governance bridge, while v0.7 adds the
full `07_interpretability_model_risk` module and upgrades shared governance
outputs to production-pattern readiness.

## User Stories

1. As a learner, I want to explain why a specific alert was generated, so that scores are not black boxes.
2. As a learner, I want to compare rule, model, graph, and case evidence, so that I can separate signal types.
3. As a learner, I want feature importance and partial-dependence-style explanations, so that model behavior is inspectable.
4. As a learner, I want optional SHAP examples where practical, so that I can see common explainability tooling without making it mandatory.
5. As a learner, I want threshold exercises tied to alert capacity and costs, so that tuning reflects operational tradeoffs.
6. As a learner, I want false-positive concentration analysis, so that I can identify where alerts burden specific segments or workflows.
7. As a learner, I want model documentation templates, so that assumptions, limits, data lineage, and monitoring needs are explicit.
8. As a learner, I want governance memo exercises, so that technical outputs become stakeholder-readable summaries.
9. As a maintainer, I want explanation utilities to be deterministic or tolerance-bounded, so that tests are reliable.
10. As a reviewer, I want governance language to remain educational and not imply certification or legal advice.

## Implementation Decisions

- Build on evaluation utilities from v0.1 and model outputs from v0.3, v0.4, and v0.6.
- Add explanation utilities only where they can be tested and documented clearly.
- Keep SHAP or heavier dependencies optional unless the team accepts the dependency cost.
- Add model documentation and governance memo templates.
- Upgrade `03_alert_governance` so it remains consistent with the deeper v0.5
  case layer, v0.6 graph evidence, and v0.7 model-risk templates.
- Connect every interpretability example to a **Detection pattern**, model output, alert decision, or case outcome.
- Teach threshold selection through alert volume, precision, recall, investigation capacity, false-positive cost, and missed-fraud cost.
- Add regulatory source notes for model risk, explainability, validation, documentation, and monitoring.

## Testing Decisions

- Test threshold summaries and cost curves against known inputs.
- Test explanation output shapes, required fields, and deterministic or tolerance-bounded values.
- Test model documentation templates for required sections.
- Test governance memo notebooks on tiny data.
- Test publication language guardrails for governance content.
- Regression-test prior notebooks that feed model outputs into v0.7.

## Out Of Scope

- Formal model registry or production approval workflow.
- Full fairness curriculum beyond segment concentration and review prompts.
- Automated governance decisions.
- Real regulatory certification.
- Mandatory SHAP if dependency or runtime cost is too high.

## Cross-Version Links

- v0.7 consumes v0.3/v0.4 model outputs, v0.5 case narratives, and v0.6 graph evidence.
- v0.7 explicitly deepens the v0.1 `03_alert_governance` module and adds the
  full `07_interpretability_model_risk` module.
- v0.8 should reuse thresholds, model docs, explanations, and monitoring requirements from v0.7.
- v0.9 capstone should require learners to explain and document model limitations.
- v1.0 should use v0.7 as the governance standard for core curriculum models.
