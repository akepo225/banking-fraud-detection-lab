# PRD: Banking Fraud Detection Lab v0.6

Status: draft for human review.

## Problem Statement

Fraud often appears through relationships rather than isolated rows. After v0.5,
learners can connect alerts to cases and regulatory context, but they still need
to model networks of **Partner**, **Client**, **User**, **Banking relationship**,
accounts, beneficiaries, devices, counterparties, transactions, alerts, and cases.

The risk is that graph analytics either arrives too early as infrastructure or
too late as an unrelated advanced topic. v0.6 should introduce graph thinking as
an extension of the existing tabular data model and case layer.

## Solution

Add a NetworkX-first graph and network analytics module. Learners should build
graphs from generated tables, compute explainable network features, connect those
features to **Detection patterns**, and use them in private-banking and
digital-banking investigation exercises. Optional Neo4j export can exist only as
a non-required path.

v0.6 delivers the `06_graph_network_analytics` module.

## User Stories

1. As a learner, I want to build a graph from existing generated tables, so that I do not need a separate graph-only dataset.
2. As a learner, I want node and edge conventions, so that graph construction is reproducible.
3. As a learner, I want private-banking graph exercises for beneficial ownership, shared counterparties, related entities, and circular movement.
4. As a learner, I want digital-banking graph exercises for mule rings, shared devices, shared beneficiaries, and pass-through clusters.
5. As a learner, I want connected components, degree, centrality, community, path length, and bridge-node features, so that graph signals are explainable.
6. As a learner, I want graph features linked to alerts and cases, so that network evidence supports investigation rather than replacing it.
7. As a learner, I want small graph examples first, so that network metrics are understandable before larger examples.
8. As a learner, I want graph visualizations that support reasoning, so that charts are not decorative.
9. As a maintainer, I want deterministic graph construction tests, so that graph outputs do not drift.
10. As a reviewer, I want optional graph database exports to remain optional, so that setup stays lightweight.

## Implementation Decisions

- Use NetworkX as the required graph layer.
- Define graph node types for **Partner**, **Client**, **User**, **Banking relationship**, account, beneficiary, device, transaction, alert, case, and **Detection pattern** where useful.
- Define edge types for ownership, authorization, login/session, payment, counterparty, shared-device, alert, case, and scenario relationships.
- Add graph feature utilities with stable, documented inputs and outputs.
- Add notebooks for graph construction, feature extraction, and graph-based fraud-ring detection.
- Keep graph examples tied to v0.5 cases and v0.3/v0.4 track signals.
- Keep Neo4j export optional and outside the core learning path.

## Testing Decisions

- Test deterministic node and edge construction from seeded data.
- Test expected graph metrics on small known fixtures.
- Test graph feature joins back to tabular **Progressive data views**.
- Test private-banking and digital-banking graph examples separately.
- Smoke-test graph notebooks on tiny data.
- Regression-test v0.3, v0.4, and v0.5 notebooks after graph additions.

## Out Of Scope

- Mandatory Neo4j, graph database deployment, or graph streaming.
- Deep learning on graphs.
- Production graph scoring services.
- Replacing tabular feature engineering.
- New regulatory claims specific to graph outputs.

## Cross-Version Links

- v0.6 depends on v0.2 entity stability, v0.3 private-banking structures, v0.4 digital structures, and v0.5 case context.
- v0.7 should use graph features as explainable evidence in model-risk and governance exercises.
- v0.8 should monitor graph-derived scores and alert volumes.
- v0.9 and v1.0 should include graph work only after learners understand tabular foundations.

## Self-Review Checklist

- The PRD keeps NetworkX required and Neo4j optional.
- The PRD builds graph views from the canonical data model.
- The PRD ties graph features to **Detection patterns** and the **Alert lifecycle**.
- The PRD stages graph complexity through **Progressive data views**.
- The PRD does not make graph analytics a prerequisite for earlier modules.
