# PRD: Banking Fraud Detection Lab v0.9

Status: draft for human review.

## Problem Statement

By v0.8, the repo should have foundations, two deep tracks, case/regulatory
skills, graph analytics, interpretability, governance, and production-pattern
monitoring. The remaining problem before v1.0 is integration. Learners need a
single capstone path that proves the pieces work together, and maintainers need a
public beta review point that identifies release-blocking gaps before the core
curriculum is declared complete.

## Solution

Deliver a capstone and public beta release. The capstone should guide learners
from scenario brief through synthetic data generation, SQL features, modeling,
alert review, graph or case evidence where appropriate, interpretability,
monitoring outputs, governance memo, and presentation. The beta should also add
release-readiness checks, troubleshooting docs, and explicit review boundaries.

## User Stories

1. As a learner, I want an end-to-end capstone, so that I can apply the full lab workflow without hidden maintainer knowledge.
2. As a learner, I want private-banking and digital-banking capstone variants, so that both **Fraud detection tracks** are represented.
3. As a learner, I want hidden scenario answer keys, so that investigation work is not solved by inspecting labels.
4. As a learner, I want a scenario brief, so that the analytic task starts from business context.
5. As a learner, I want to generate data, load SQLite, write SQL features, score models, review alerts, and document limitations, so that the capstone exercises the whole path.
6. As a learner, I want a rubric, so that expectations for SQL quality, fraud reasoning, model evaluation, alert interpretation, governance, and communication are clear.
7. As a learner, I want a capstone presentation template, so that findings can be communicated consistently.
8. As a reviewer, I want a beta checklist, so that release blockers are separated from polish.
9. As a maintainer, I want notebook and command validation, so that the beta is reproducible from a clean setup.
10. As a maintainer, I want documentation for common setup and execution failures, so that learners can recover independently.
11. As a reviewer, I want public-facing terminology audited, so that old or ambiguous framing does not return.
12. As a future contributor, I want issue templates and contribution guidance, so that post-beta feedback is structured.

## Implementation Decisions

- Integrate existing modules rather than adding a large new feature family.
- Provide private-banking and digital-banking capstone variants with comparable structure but different **Detection patterns**.
- Keep capstone answer keys protected by default.
- Add a capstone rubric and presentation template focused on analytic reasoning and limitations.
- Add README polish with screenshots, schema previews, or concise output previews where useful.
- Add public issue templates and contributor guidance only after publication boundaries are reviewed.
- Treat v0.9 as beta: release-blocking issues should be visible and triaged before v1.0.

## Testing Decisions

- Smoke-test capstone notebooks or scripts on tiny data.
- Test protected answer-key exclusion for capstone datasets.
- Test documented capstone commands from a clean setup where feasible.
- Test links and references across README, notebook guide, SQL guide, case library, regulatory index, and capstone docs.
- Test terminology guardrails for public-facing files.
- Rerun the full quality gate suite before marking beta ready.

## Out Of Scope

- Declaring v1.0 complete.
- Adding new advanced tracks.
- New **Detection patterns**, new scenario families, or new data model entities
  not already present in v0.2 through v0.8.
- Heavy infrastructure beyond v0.8 production patterns.
- Real institution or client data, real-bank relationship claims, or real event reconstruction.
- A hosted learning platform or grading service.

## Cross-Version Links

- v0.9 consumes every core release from v0.2 through v0.8.
- v0.9 should surface integration gaps that v1.0 must resolve.
- v1.0 should harden the beta path rather than expanding feature scope.
- v1.1 through v1.4 should remain out of the capstone unless optional advanced variants are explicitly added later.

## Self-Review Checklist

- The PRD makes v0.9 an integration and beta release, not a new feature expansion.
- The PRD includes both private-banking and digital-banking capstone variants.
- The PRD keeps protected answer keys out of learner-facing outputs.
- The PRD includes release-blocking quality gates and manual review items.
- The PRD creates a clear hand-off to v1.0.
