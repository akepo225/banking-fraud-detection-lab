# PRD: Banking Fraud Detection Lab v1.1

Status: draft for human review.

## Problem Statement

v1.0 completes the public core curriculum, but advanced learners also need to
study digital-asset fraud where banking activity intersects with fiat on/off
ramps, wallet withdrawals, exchange accounts, scams, mule behavior, vendor risk
scores, and graph-like movement. This work should extend the lab without making
digital assets part of the required core.

## Solution

Add an optional digital-assets and crypto-fraud extension. v1.1 should introduce
synthetic wallets, exchanges, token transfers, chain-risk events, fiat on-ramp and
off-ramp activity, crypto-withdrawal alerts, and governance notes. It should stay
anchored in banking fraud by connecting wallet activity back to **Client**,
**User**, **Partner**, **Banking relationship**, accounts, transactions, cases,
and the **Alert lifecycle**.

## User Stories

1. As a learner, I want fiat on/off-ramp data linked to bank accounts, so that digital-asset risk stays connected to banking behavior.
2. As a learner, I want wallet and exchange entities, so that crypto movement can be analyzed without real blockchain ingestion.
3. As a learner, I want scam-to-crypto and mule-to-crypto scenarios, so that v0.4 digital patterns extend naturally.
4. As a learner, I want wallet-risk, chain-hop, velocity, and destination-risk features, so that crypto **Detection patterns** are explainable.
5. As a learner, I want graph-style wallet movement views, so that v0.6 network skills transfer to digital assets.
6. As a learner, I want vendor or **Partner** risk scores treated as signals, so that third-party data is not mistaken for ground truth.
7. As a learner, I want false-positive examples from legitimate digital-asset activity, so that review remains realistic.
8. As a learner, I want governance notes for digital-asset risk models, so that explainability and data-source limits are visible.
9. As a maintainer, I want v1.1 to be optional, so that v1.0 remains stable.
10. As a reviewer, I want synthetic-only boundaries, so that no real wallet or client data is introduced.

## Implementation Decisions

- Add v1.1 as an optional advanced module outside the v1.0 core path.
- Extend the data model with wallet, exchange, chain transaction, token transfer, on-ramp, off-ramp, and chain-risk event tables or views.
- Reuse **Client**, **User**, **Partner**, **Banking relationship**, **Detection pattern**, **Progressive data view**, and **Alert lifecycle** terminology.
- Keep chain-risk scores explainable and limited.
- Build simple tabular views before graph or multi-hop wallet views.
- Add case and regulatory source notes only as educational anchors.

## Testing Decisions

- Test deterministic crypto scenario generation and low-prevalence ranges.
- Test joins between bank transactions, sessions, wallet events, and risk scores.
- Test progressive views so early exercises do not expose advanced answer-like graph data.
- Test detection features for wallet novelty, velocity, destination risk, and rapid liquidation.
- Regression-test v1.0 core outputs and notebooks.
- Pass the full v1.0 test suite and all v1.0 core notebook smoke tests without
  requiring changes to the v1.0 learner path.

## Out Of Scope

- Real blockchain ingestion.
- Real wallet attribution or sanctions-list integration.
- Live custody, trading, or transfer functionality.
- Making crypto a required v1.0 path.
- Claims of regulatory or vendor accuracy.

## Cross-Version Links

- v1.1 depends on v1.0 core data, graph, governance, and monitoring foundations.
- v1.2 can reuse event-risk patterns for brokerage activity.
- v1.3 can reuse scam and coercion scenarios in communication monitoring.
- v1.4 can optionally route crypto events through advanced infrastructure.

## Self-Review Checklist

- The PRD keeps digital assets optional and banking-anchored.
- The PRD treats external risk scores as signals with limitations.
- The PRD uses **Progressive data views** before wallet graph complexity.
- The PRD protects v1.0 from regression.
- The PRD avoids real-chain or real-client data dependencies.
