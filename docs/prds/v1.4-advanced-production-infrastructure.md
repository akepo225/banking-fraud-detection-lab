# PRD: Banking Fraud Detection Lab v1.4

Status: draft for human review.

## Problem Statement

v0.8 teaches local production patterns, and v1.1 through v1.3 add advanced event
types such as crypto, brokerage, and communication signals. Advanced learners may
then need a production-infrastructure extension that demonstrates event contracts,
streaming or micro-batch scoring, feature pipelines, model serving, graph updates,
dashboards, drift detection, audit trails, rollback, and degraded-mode behavior.

The risk is that infrastructure complexity overwhelms fraud-detection learning or
becomes mandatory for earlier modules.

## Solution

Add an optional advanced production infrastructure module. v1.4 should provide a
local-first, opt-in environment for production-like fraud detection architecture
while preserving the v1.0 core curriculum and the v1.1-v1.3 advanced modules as
independent learning paths.

## User Stories

1. As a learner, I want to run a simulated event pipeline, so that batch, micro-batch, and streaming tradeoffs are visible.
2. As a learner, I want stable event contracts for transactions, wallet events, orders, communications, graph updates, scores, and alerts, so that pipelines are understandable.
3. As a learner, I want a common scoring interface for rules, ML models, graph features, and NLP signals, so that multiple **Detection patterns** can be compared.
4. As a learner, I want feature lineage from source event to score to alert decision, so that auditability is explicit.
5. As a learner, I want latency, throughput, alert volume, data quality, and drift monitoring, so that operational health is visible.
6. As a learner, I want rollback and degraded-mode examples, so that model or **Partner** signal failures can be handled.
7. As a learner, I want dashboards or reports that explain operational health, so that monitoring can be communicated.
8. As a learner, I want environment separation concepts, so that development, validation, and production-like configuration are distinct.
9. As a maintainer, I want advanced services optional, so that v1.0 setup stays simple.
10. As a reviewer, I want no claim of real bank production readiness, so that the educational scope remains clear.

## Implementation Decisions

- Keep v1.4 optional and advanced.
- Build on v0.8 production-pattern concepts rather than replacing them.
- Provide local-first defaults and optional heavier components only where justified.
- Define stable event contracts for banking transactions, crypto events, brokerage events, communication events, graph updates, model scores, alert decisions, reviewer actions, and audit events.
- Add a common scoring interface across rule, model, graph, and NLP signals.
- Add model deployment metadata and registry-like concepts without requiring cloud services.
- Add failure scenarios for delayed events, malformed data, missing **Partner** signals, feature skew, model degradation, and alert backlog.
- Keep v1.0 notebooks and data paths independent from v1.4 services.

## Testing Decisions

- Test event schemas and backward-compatible synthetic generation.
- Test local startup and teardown for optional services if added.
- Test scoring latency and throughput against local educational targets.
- Test alert lifecycle transitions in batch, micro-batch, and streaming-like modes.
- Test failure scenarios, rollback, malformed events, missing partner signals, and delayed events.
- Test audit trail completeness from source event to alert decision.
- Regression-test v1.0 core and v1.1-v1.3 optional modules without v1.4 enabled.
- Pass the full v1.0 test suite and all v1.0 core notebook smoke tests without
  requiring changes to the v1.0 learner path.

## Out Of Scope

- Cloud deployment as a hard requirement.
- Real bank infrastructure integration.
- Real-time production SLA guarantees.
- Enterprise identity management.
- Rebuilding all modules as microservices.
- Making advanced infrastructure mandatory for learners.

## Cross-Version Links

- v1.4 depends on v1.0 production-pattern, governance, graph, and alert foundations.
- v1.4 can optionally integrate event types from v1.1, v1.2, and v1.3.
- v1.4 should not alter the v1.0 core path or require advanced modules to be installed.
- Future releases can extend v1.4 toward resilience, governance automation, or deployment templates.

## Self-Review Checklist

- The PRD keeps infrastructure optional and advanced.
- The PRD protects v1.0 from service dependencies.
- The PRD keeps fraud-detection learning outcomes central.
- The PRD defines event contracts before services.
- The PRD includes failure-mode, audit, drift, and rollback testing.
