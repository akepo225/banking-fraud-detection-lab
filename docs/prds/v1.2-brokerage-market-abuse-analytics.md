# PRD: Banking Fraud Detection Lab v1.2

Status: draft for human review.

## Problem Statement

The core curriculum covers banking fraud detection but does not deeply address
brokerage, securities order behavior, suitability anomalies, account takeover
trading, or market-abuse-style analytics. Advanced learners need a structured
extension that shows how trading and investment activity intersects with fraud
signals in private-banking and digital-banking contexts.

## Solution

Add an optional brokerage and market-abuse analytics module. v1.2 should extend
the synthetic model with instruments, orders, executions, holdings, suitability
attributes, market events, restricted-list concepts, and trading alerts. The
module should teach order and execution features, account takeover trading,
coordinated behavior, suspicious product changes, and governance limits.

## User Stories

1. As a learner, I want instruments, orders, executions, and holdings, so that brokerage analytics can be modeled.
2. As a learner, I want trading linked to **Client**, **User**, **Partner**, accounts, and **Banking relationship**, so that trading risk remains connected to banking context.
3. As a learner, I want account takeover trading examples, so that digital access and brokerage behavior can be analyzed together.
4. As a learner, I want suitability and profile-drift examples, so that private-banking investment behavior has context.
5. As a learner, I want coordinated or wash-like synthetic patterns, so that network analytics can extend to trading.
6. As a learner, I want time-window features around sensitive market events, so that temporal logic matters.
7. As a learner, I want sparse-label evaluation, so that ambiguous trading alerts are not treated as perfect truth.
8. As a learner, I want governance notes that distinguish fraud analytics from broader market surveillance, so that scope remains clear.
9. As a maintainer, I want v1.2 to be optional and additive, so that v1.0 remains stable.
10. As a reviewer, I want synthetic market events only, so that no live market data dependency is introduced.

## Implementation Decisions

- Add v1.2 as an optional advanced module outside the v1.0 core path.
- Add synthetic entities for instruments, orders, executions, positions, holdings, suitability attributes, market events, and trading alerts.
- Reuse the shared **Alert lifecycle**, **Detection pattern**, graph, governance, and monitoring patterns.
- Support Alpine Crest Private Bank investment-advisory context and NovaBank Digital self-directed brokerage context.
- Keep market-abuse examples educational and synthetic.
- Use **Progressive data views** from order summaries to execution detail to cross-client graphs.

## Testing Decisions

- Test order/execution consistency, fills, positions, and holdings.
- Test temporal joins around synthetic market events.
- Test account takeover trading and profile-drift feature calculations.
- Test graph relationships among **Client**, **User**, instruments, accounts, and devices.
- Test alert lifecycle transitions for brokerage alerts.
- Regression-test v1.0 and v1.1 optional paths where relevant.
- Pass the full v1.0 test suite and all v1.0 core notebook smoke tests without
  requiring changes to the v1.0 learner path.

## Out Of Scope

- Real trading connectivity or live market data.
- Investment advice engines.
- Production surveillance certification.
- Making brokerage analytics mandatory for v1.0 learners.
- Replacing banking fraud as the main curriculum purpose.

## Cross-Version Links

- v1.2 depends on v1.0 core and may reuse v1.1 event-risk and graph patterns.
- v1.3 can use brokerage communications and suspicious-intent text as optional NLP scenarios.
- v1.4 can route orders and executions through advanced production infrastructure.

## Self-Review Checklist

- The PRD keeps brokerage as an advanced extension.
- The PRD connects trading to banking entities and the **Alert lifecycle**.
- The PRD distinguishes fraud, suspicious trading analytics, and educational governance.
- The PRD avoids live market or trading dependencies.
- The PRD keeps v1.0 stable.
