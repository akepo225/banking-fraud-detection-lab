## context.md — your survival anchor across compactions

Create `context.md` at the worktree root on the first turn and KEEP IT UPDATED.
It is your first read after every context compaction. It MUST contain:

- **Current focus**: issue number + wave + one-line status.
- **Done log**: `#issue → PR# → merged (sha)?` for every closed item.
- **Live todo list**: remaining issues by wave, next concrete action.
- **Decisions/blockers**: HITL decisions taken, reviewer asks awaiting reply,
  anything an incoming-you must not redo or must not forget.
- **Reuse map**: the v0.7/v0.6 symbols each remaining issue must consume (below).

Update context.md after every commit, every PR open, every review comment, and
every merge. If you are about to act and context.md is stale, update it first.

## Per-issue loop

### Phase 1 — Discovery & Analysis
1. `gh issue view {{ISSUE_NUMBER}} --comments` — full body + threads.
2. `gh issue view 51 --comments` — the PRD (contextual ticket).
3. Treat any `coderabbitai` plan in the comments as **guidance only**. Your
   codebase analysis is the primary directive; reconcile the plan against actual
   code before adopting any step.

### Phase 2 — Planning
- Synthesize PRD #51 + issue #{{ISSUE_NUMBER}} + codebase reality into one plan.
- Verify each step against current code (conventions beat the issue text).
- Record the plan in the live todo list + context.md before coding.

### Phase 3 — Execution Setup
- `git status` clean; `git fetch origin main`.
- Create a **fresh per-issue branch** `feat/issue-{{ISSUE_NUMBER}}-implementation`
  off `origin/main` in an isolated worktree (NOT in the primary checkout, which
  sits on an unrelated branch — never touch it).

### Phase 4 — Implementation + Pre-commit Gate
- Match existing style/naming/architecture. Delegate independent components to
  sub-agents; validate outputs before staging.
- **Pre-commit gate — commit only pristine code:**
  1. **Self-review**: re-read every changed file. Bugs, typos, missing error
     handling, hardcoded values, leftover debug, incomplete TODOs.
  2. **CodeRabbit local review**: `coderabbit review` (or local equivalent) on
     the staged diff. Fix every legitimate finding before committing.
  3. **Tests + lint + docstrings**: `uv run ruff check .` and `uv run pytest`
     must pass; inspect every new/modified public function/class/module for a
     docstring and add missing ones.
  4. Only then commit + push.

### Phase 5 — Delivery
1. Install `gh-pr-context` if not on PATH:
   `curl -fsSL https://raw.githubusercontent.com/akepo225/gh-pr-context/master/install.sh | bash`
2. Load the `deliver` skill: `npx skills@latest add akepo225/gh-pr-context`.
3. Open a PR targeting `main`. Description: what/why/how-tested, link
   `Closes #{{ISSUE_NUMBER}}`.
4. Iterate using `gh-pr-context comments`, `--since last-commit`, `status`,
   `logs`. (Fallback: `gh api`; never `gh pr view` for nested comments.)
5. **Evaluating review comments**: do a brief codebase analysis before acting.
   Implement only comments with grounded added value; dismiss stylistic /
   speculative ones with a clear justification in the thread. No reflexive changes.
6. Re-run the Phase 4 gate before every fix-push.
7. Tag reviewers `{{REVIEWER_TAGS}}`; wait a few minutes for replies.

### Phase 6 — Issue done conditions (all must hold)
- [ ] Every acceptance criterion in #{{ISSUE_NUMBER}} met.
- [ ] CI fully green.
- [ ] All reviewer comments addressed or dismissed with justification.
- [ ] Reviewers confirm mergeable.
- [ ] Out-of-scope items filed as follow-up issues.
- [ ] Deferred code carries `// TODO(#<n>): ...` linking its issue.
- [ ] context.md updated; move to next unblocked issue.
