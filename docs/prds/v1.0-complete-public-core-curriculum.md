# PRD: Banking Fraud Detection Lab v1.0

Status: draft for human review.

## Problem Statement

After v0.9 beta, the project needs a stable complete public core curriculum. The
main problem is not feature quantity; it is coherence, reproducibility,
documentation, quality gates, and learner independence. v1.0 must prove that the
core path from foundations to capstone can be completed with the repository alone.

## Solution

Ship the complete public core curriculum for **Banking Fraud Detection Lab**. The
release should lock the core module sequence, run all featured notebooks from a
clean setup, validate synthetic data and SQLite exercises, confirm case and
regulatory source discipline, and publish consistent documentation, disclaimers,
licenses, contribution guidance, and roadmap boundaries.

## User Stories

1. As a new learner, I want a clear start path, so that I can complete the core curriculum without private guidance.
2. As a learner, I want modules `00` through `09` to run in order, so that the curriculum has a coherent sequence.
3. As a learner, I want the **Realistic synthetic data model** explained once and reused throughout, so that concepts accumulate.
4. As a learner, I want **Progressive data views**, so that each module exposes the right level of complexity.
5. As a learner, I want both Alpine Crest Private Bank and NovaBank Digital tracks to be complete, so that private-banking and digital-banking fraud detection are first-class.
6. As a learner, I want cases, regulatory notes, graph analytics, governance, monitoring, and capstone work linked to earlier models, so that modules are connected.
7. As a maintainer, I want all notebooks and tests to run from a clean setup, so that release quality is verifiable.
8. As a maintainer, I want tiny sample data committed and larger data generated locally, so that the repo stays lightweight.
9. As a reviewer, I want docs, licenses, disclaimers, roadmap, and ADRs to be consistent, so that public framing is credible.
10. As a contributor, I want clear extension boundaries, so that post-1.0 additions do not destabilize the core curriculum.
11. As a release reviewer, I want a v1.0 checklist, so that acceptance criteria are explicit.

## Implementation Decisions

- Freeze the public core module list from `00_foundations` through `09_capstone`.
- Prioritize fixes, documentation, test hardening, and learner flow over new functionality.
- Triage v0.9 beta findings as v1.0 blockers or post-v1.0 follow-ups. v1.0 does
  not accept new features or new scenario families.
- Keep SQLite first-class and local-first.
- Ensure each featured notebook has a corresponding smoke test or documented execution path.
- Ensure each module links to relevant cases, regulatory notes, and quality gates where appropriate.
- Document extension points for new scenarios, **Detection patterns**, views, and advanced modules.
- Keep v1.1 through v1.4 outside the required core curriculum.

## Testing Decisions

- Run full tests, linting, and notebook smoke checks on a clean checkout.
- Validate deterministic sample regeneration and generated-data scale profiles.
- Validate SQLite loader and representative SQL examples.
- Validate schema/data-dictionary alignment and module view docs.
- Validate case metadata and regulatory note structure.
- Validate publication-language guardrails.
- Add a v1.0 release checklist that maps acceptance criteria to evidence.

## Out Of Scope

- New advanced tracks.
- Mandatory PostgreSQL, Neo4j, Kafka, stream processing, dashboards, or deployed APIs.
- Real institution data or real-bank relationship claims.
- Hosted learning platform, grading system, or instructor portal.
- Full beginner programming curriculum.

## Cross-Version Links

- v1.0 consumes v0.2 through v0.9 and should resolve or explicitly defer beta findings.
- v1.1 through v1.4 must build as optional extensions on top of v1.0.
- v1.0 establishes the terminology, quality gates, and extension contract for all advanced releases.

## Self-Review Checklist

- The PRD hardens scope instead of expanding beyond beta.
- The PRD makes all core modules runnable and documented.
- The PRD keeps advanced tracks optional and out of core.
- The PRD includes release evidence for code, data, docs, notebooks, and public framing.
- The PRD gives future releases a stable extension boundary.
