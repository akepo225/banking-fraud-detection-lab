## Decision

**Ready for human v0.8 acceptance review.** The v0.8 child issues
(#200–#209) are the delivery mechanism; their PRs (#213–#222) carry the
implementation. This review (#210) is the final sign-off artifact and is not
self-merged — it opens for human sign-off. The HITL slice #200 (frozen
monitoring-table contract, PR #213) was the contract gate the rest of v0.8 built
against, and this #210 gate is the second HITL checkpoint.

v0.8 introduces the production-monitoring patterns module (`08`). Learners can
run a deterministic batch scoring pipeline, inspect a frozen five-table
monitoring contract (`score`, `threshold`, `alert_decision`, `reviewer_action`,
`audit_event`), follow the alert-decision / reviewer-action / audit-event
lineage from a score back to its Banking relationship / Client / User /
Detection pattern, walk an alert queue with operational metrics, and run drift
and data-quality checks that reuse the v0.7 governance checklist dimensions.
Two track notebooks (Alpine Crest Private Bank, NovaBank Digital) and one
track-comparison / alert-review governance notebook exercise the module
end-to-end on tiny data. Heavy real-time infrastructure (Kafka, Spark, Redis,
service orchestration, external dashboards) stays **optional / out-of-CI**
(#206), mirroring the v0.7 SHAP decision (#187) and the v0.6 Neo4j decision
(#154): the curriculum teaches the operational patterns without a mandatory
heavy dependency.

The repository remains private until the separate v0.1 publication gate is
accepted; v0.8 adds no public-release language and makes no
production-readiness or certification claim.

## Exit Criteria (PRD / ROADMAP)

- ✅ **Exit criterion 1** ("Learners can run a local batch scoring flow and
  inspect generated alerts"): the three `08` notebooks execute end-to-end via
  nbconvert on tiny data (their own smoke tests), exercising `run_batch_scoring`
  → `decide_alerts` → `inspect_alert_queue`. Enforced by
  `tests/test_v08_acceptance_review.py` (notebook presence + module presence)
  and the per-notebook smoke tests.
- ✅ **Exit criterion 2** ("Monitoring outputs are tied to governance and alert
  lifecycle concepts"): every monitoring table carries Banking-relationship /
  Client / User / Detection-pattern / Alert lineage; the audit-event vocabulary
  (`AUDIT_EVENT_TYPES`) traces the Alert lifecycle
  (`score_assigned` → `alert_decision_made` → `reviewer_action_recorded` →
  `threshold_reviewed`); drift and data-quality results reference governance
  `MONITORING_CHECKLIST_DIMENSION_IDS` (`score_drift`, `data_quality`); and the
  threshold / reviewer-action builders reuse v0.7 symbols
  (`recommend_lowest_cost_threshold`, `evaluate_alert_scores`,
  `explain_feature_family`, `build_model_documentation`,
  `build_monitoring_checklist`, `MONITORING_CHECKLIST_DIMENSIONS`). Enforced by
  `tests/test_v08_acceptance_review.py`.

## Testing Decisions (PRD)

## HITL: Real-Time-Infrastructure Dependency-Cost Decision (slice #206)

Per PRD #51: *"Keep Kafka, Spark, Redis, service orchestration, and external
dashboards optional or deferred."*

**Decision accepted: heavy real-time infrastructure stays optional and outside
CI.** Kafka / Spark / Redis / service orchestration / external dashboards would
pull in heavyweight dependencies and operational cost disproportionate to a
curriculum whose required path is local batch scoring. None of these libraries
are declared in `pyproject.toml` core or dev dependencies (and therefore are
not installed by `uv sync --extra dev` or required by CI). This mirrors the v0.7
SHAP decision (slice #187) and the v0.6 Neo4j decision (slice #154). Slice
The notes for slice #206 (`docs/advanced/realtime_infrastructure.md`) sketch
what real-time infrastructure would add later, after v1.0 is stable.
