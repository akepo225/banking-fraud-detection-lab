# PRD: Banking Fraud Detection Lab v0.8

Status: draft for human review.

## Problem Statement

The curriculum teaches data generation, features, cases, graph evidence, and
governance, but learners still need to see how fraud scoring moves toward
production-like operation. The goal is not to build real banking infrastructure.
The goal is to teach batch scoring, score tables, alert queues, monitoring,
drift checks, reviewer actions, and audit trails in a lightweight local setting.

The risk is either too little operational realism or too much mandatory
infrastructure.

## Solution

Add local-first production patterns and monitoring exercises. v0.8 should let
learners run deterministic batch scoring, write score and alert decision tables,
inspect alert queues, monitor drift and data quality, review thresholds, and tie
monitoring outputs back to governance and the **Alert lifecycle**.

v0.8 delivers the `08_production_monitoring_patterns` module.

## User Stories

1. As a learner, I want to run a batch scoring pipeline, so that notebook models become repeatable scoring flows.
2. As a learner, I want score, threshold, alert decision, reviewer action, and audit tables, so that operational data is explicit.
3. As a learner, I want to inspect alert queues and alert aging, so that the **Alert lifecycle** has operational meaning.
4. As a learner, I want to monitor alert volume, precision, recall, capacity utilization, and closure outcomes, so that quality is not only model-centric.
5. As a learner, I want drift checks for amount, geography, channel, device, relationship activity, and user behavior, so that monitoring connects to features.
6. As a learner, I want to compare Alpine Crest Private Bank and NovaBank Digital monitoring behavior, so that track differences are visible.
7. As a learner, I want to trace a monitoring anomaly back to **Client**, **User**, **Banking relationship**, and **Detection pattern** records.
8. As a learner, I want to reuse v0.7 explanations during alert review, so that monitoring and governance are connected.
9. As a maintainer, I want deterministic scoring outputs, so that production-pattern tests are stable.
10. As a reviewer, I want infrastructure language to stay educational, so that the repo does not claim production readiness.

## Implementation Decisions

- Use local batch scoring as the required production pattern.
- Add a CLI or simple command interface only if it improves repeatability.
- Represent monitoring outputs as local tables, reports, and notebooks.
- Add monitoring table schemas for scores, thresholds, alert decisions, reviewer actions, and audit events.
- Use v0.7 thresholds, model documentation, and explanation artifacts as monitoring inputs.
- Keep Kafka, Spark, Redis, service orchestration, and external dashboards optional or deferred.
- Add clear advanced notes explaining what real-time infrastructure would add later.

## Testing Decisions

- Test deterministic batch scoring outputs.
- Test monitoring table creation, schema contracts, and foreign keys.
- Test alert-volume, alert-aging, closure-distribution, and capacity metrics.
- Test drift checks on controlled synthetic shifts.
- Test reviewer-action and audit-event lineage from score to alert decision.
- Smoke-test production-pattern notebooks or CLI commands.
- Regression-test prior model and evaluation utilities.

## Out Of Scope

- Real-time streaming as a required implementation.
- Deployed APIs, authentication, authorization, or real operations security.
- Enterprise observability stack.
- Real bank integration.
- Formal runbooks for real institutions.

## Cross-Version Links

- v0.8 consumes v0.7 governance artifacts and thresholds.
- v0.8 consumes v0.3/v0.4 models and v0.6 graph features as scoring inputs.
- v0.9 capstone should require learners to inspect monitoring and alert-review outputs.
- v1.0 should include v0.8 as the production-pattern module, not as an infrastructure product.
- v1.4 can later make heavier infrastructure optional after v1.0 is stable.

## Self-Review Checklist

- The PRD teaches production patterns without claiming production readiness.
- The PRD keeps local-first batch scoring as the required path.
- The PRD ties monitoring to the **Alert lifecycle**, not generic ML dashboards.
- The PRD reuses v0.7 model-risk artifacts.
- The PRD avoids mandatory heavy infrastructure.
