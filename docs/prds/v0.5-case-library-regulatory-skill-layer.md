# PRD: Banking Fraud Detection Lab v0.5

Status: draft for human review.

## Problem Statement

By v0.4, the repo should have hardened foundations plus deeper private-banking
and digital-banking tracks. The next gap is interpretation skill. Learners need
to connect synthetic alerts and **Detection patterns** to public-source cases,
source discipline, regulatory context, evidence quality, and investigation
reasoning without treating the material as legal or compliance advice.

The risk is that cases and regulatory notes stay as passive reading material
instead of becoming a practical skill layer tied to notebooks, alerts, and
features.

## Solution

Turn the seeded case library and regulatory source index into a structured skill
layer. v0.5 should add pattern-first case templates, metadata validation, source
quality rubrics, linked exercises, regulatory learning prompts, and case-narrative
tasks for Alpine Crest Private Bank and NovaBank Digital.

v0.5 is a cross-cutting enrichment release, not a standalone numbered core
module. It strengthens modules `00` through `05` and creates reusable source
discipline for later modules.

## User Stories

1. As a learner, I want to read a public-source case and extract **Detection patterns**, so that case reading turns into analytic work.
2. As a learner, I want to separate source facts from interpretation, so that I avoid overclaiming what happened.
3. As a learner, I want source-quality rubrics, so that regulator, court, official disclosure, journalism, and industry sources are weighted appropriately.
4. As a learner, I want cases linked to notebooks and data views, so that reading connects to hands-on exercises.
5. As a learner, I want private-banking source packs tied to Alpine Crest Private Bank features, so that cases inform relationship-led analytics.
6. As a learner, I want digital-banking source packs tied to NovaBank Digital features, so that cases inform scam, mule, ATO, and payment analytics.
7. As a learner, I want regulatory notes organized by analytics use, so that controls, documentation, explainability, alert handling, and governance become practical.
8. As a learner, I want case-narrative exercises, so that I can write concise investigation reasoning from evidence.
9. As a contributor, I want templates and validation rules, so that new cases keep source discipline.
10. As a reviewer, I want HITL markers for sensitive framing, so that publication-risk areas receive human review.
11. As a maintainer, I want case metadata to be machine-readable, so that later modules can index by track, pattern, product, geography, source type, and linked module.

## Implementation Decisions

- Keep case and regulatory material as educational mappings, not advice.
- Use structured front matter for case metadata and regulatory notes.
- Require links from each core module to at least one case or source note.
- Require every case pack to include at least one notebook exercise, SQL task, or
  feature-interpretation task that produces learner output.
- Add templates for summary, sources, facts versus interpretation, **Detection patterns**, likely data signals, exercises, regulatory hooks, limitations, and HITL review markers.
- Organize regulatory notes by learning use rather than by law name alone.
- Keep case exercises tied to existing synthetic data and **Progressive data views**.
- Do not add new complex generator scenarios solely to fit a case.

## Testing Decisions

- Validate required case metadata and required sections.
- Validate official-source links where notes claim regulatory anchors.
- Validate linked modules point to existing notebooks or docs.
- Validate no imperative legal/compliance instruction wording.
- Validate source packs include limitations and avoid reconstruction claims.
- Smoke-test any notebook exercises added for case interpretation.
- Add documentation checks for source-quality rubric and contributor guidance.

## Out Of Scope

- Full legal or compliance training.
- Reconstructing real events in synthetic data.
- Large new scenario families unrelated to v0.3 and v0.4.
- Graph analytics as a dedicated module.
- Model interpretability and production monitoring.
- A standalone numbered module slot unless a later roadmap change explicitly adds
  one.

## Cross-Version Links

- v0.5 consumes the track depth from v0.3 and v0.4.
- v0.5 enriches existing modules instead of creating a separate numbered module.
- v0.6 should use v0.5 case patterns as narrative entry points for graph evidence.
- v0.7 should reuse case limitations and regulatory hooks in governance templates.
- v0.8 should reuse case lifecycle examples for operational monitoring scenarios.
- v0.9 and v1.0 should treat v0.5 source discipline as a publication-quality gate.

- The PRD does not expand generator scope just to imitate public cases.
