# PRD: Banking Fraud Detection Lab v0.3

Status: draft for human review.

## Problem Statement

After v0.2 hardens the shared foundation, the curriculum needs a deeper
**Private-banking fraud detection** track. v0.1 contains a baseline Alpine Crest
Private Bank transaction-fraud module, but the envisioned scope requires richer
relationship context, high-value transaction behavior, beneficial ownership,
authorized signatories, relationship-manager responsibility, and private-banking
case interpretation.

The risk is that private-banking modules become generic transaction monitoring
unless **Partner**, **Client**, **Banking relationship**, account, counterparty,
and relationship-manager context drive the learning path.

## Solution

Deliver a coherent Alpine Crest Private Bank track that extends the v0.2
foundation with private-banking-specific **Detection patterns**, feature views,
SQL exercises, notebooks, case-library entries, and regulatory notes. The track
should teach learners to detect and explain anomalous high-value movement in a
relationship-driven banking environment.

v0.3 does not replace the v0.1 `01_private_banking_transaction_fraud` baseline.
It adds the full `04_private_banking_feature_engineering` module that builds on
that baseline and turns it into a deeper private-banking track.

## User Stories

1. As a learner, I want Alpine Crest Private Bank scenarios, so that private-banking fraud detection has a consistent fictional setting.
2. As a learner, I want relationship-manager assignment history, so that I can reason about responsibility at the time of activity.
3. As a learner, I want beneficial ownership and authorized signatory joins, so that complex private-banking relationships are visible.
4. As a learner, I want account and transaction typologies for wires, FX, fees, and investment-related flows, so that high-value movement feels realistic.
5. As a learner, I want amount-to-AUM and amount-to-relationship-baseline features, so that raw amount rules are not the whole model.
6. As a learner, I want new counterparty, off-hours, cross-border, and velocity features, so that **Detection patterns** are tied to explainable signals.
7. As a learner, I want false-positive examples from legitimate high-net-worth behavior, so that alert review is realistic.
8. As a learner, I want threshold tuning with alert capacity, so that model decisions reflect investigation constraints.
9. As a learner, I want private-banking case notes, so that public-source learning anchors connect to analytic features.
10. As a learner, I want Swiss regulatory source notes connected to analytics decisions, so that governance is part of the track.
11. As a maintainer, I want private-banking generation to remain deterministic, so that tests and notebooks are reproducible.
12. As a reviewer, I want every private-banking feature to map to a **Detection pattern**, so that the track is not a feature dump.

## Implementation Decisions

- Build v0.3 as an extension of v0.2 entities and views, not a separate schema.
- Use Alpine Crest Private Bank only for private-banking scenarios.
- Make relationship context more important than channel or device context.
- Add private-banking **Detection pattern** metadata and feature definitions.
- Define a track convention contract for v0.3 and v0.4 covering feature naming,
  notebook layout, **Detection pattern** metadata, evaluation output format, and
  case/regulatory link structure.
- Add a private-banking feature library with SQL and Python usage paths.
- Add notebooks for feature engineering, supervised baseline modeling, threshold tuning, and alert interpretation.
- Use case and regulatory notes to frame analytic choices, not to provide legal or compliance advice.
- Keep labels protected where the learner should investigate rather than inspect answers.

## Testing Decisions

- Test private-banking scenario prevalence ranges and deterministic outputs.
- Test referential integrity across **Partner**, **Client**, **Banking relationship**, accounts, relationship managers, signatories, beneficiaries, transactions, alerts, and cases.
- Test feature calculations for amount ratios, novelty, velocity, cross-border behavior, and relationship-manager concentration.
- Test SQL examples against SQLite.
- Test that private-banking metadata and notebook structure follow the shared
  track convention contract.
- Smoke-test private-banking notebooks on tiny data.
- Regression-test v0.2 foundation notebooks and data views after private-banking additions.

## Out Of Scope

- NovaBank Digital track expansion.
- Device fingerprinting, session-risk, account takeover, or scam-to-mule expansion beyond shared fields.
- Graph network analytics as a dedicated module.
- Production monitoring and scoring services.
- Formal regulatory workflow simulation.

## Cross-Version Links

- v0.3 consumes v0.2 temporal and schema hardening.
- v0.3 should establish track-extension conventions that v0.4 mirrors for digital banking.
- v0.3 adds `04_private_banking_feature_engineering` on top of the v0.1
  `01_private_banking_transaction_fraud` baseline.
- v0.5 should reuse v0.3 private-banking cases and regulatory hooks.
- v0.6 should reuse private-banking ownership, signatory, counterparty, and relationship-manager structures as graph inputs.
- v0.7 and v0.8 should reuse v0.3 models and thresholds for governance and monitoring examples.

## Self-Review Checklist

- The PRD makes private banking distinct through relationship context, not just transaction amount.
- Every feature family is tied to a named **Detection pattern**.
- The PRD does not introduce digital-banking-specific work prematurely.
- The PRD preserves the shared **Alert lifecycle** from v0.2.
- The PRD has enough tests to protect features, notebooks, and data integrity.
- The PRD defines a track convention contract before v0.4 implements the sibling
  digital track.
